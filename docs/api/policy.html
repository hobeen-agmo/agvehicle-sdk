자격 · 실행레벨(Tier)

이 문서는 여러 앱이 같은 제어 신호를 동시에 쥐려 할 때 무엇이 허용되고 무엇이 차단되는지를 정하는 두 축을 설명한다. 자격 선언은 이 앱이 어떤 신호를 쓰는지를 매니페스트 meta-data에 선언하고, 실행레벨 Tier는 누가 누구를 선점할 수 있는지를 정하며, 이 둘이 정적 검사와 런타임 중재로 이중 방어를 이룬다.

자격 선언

매니페스트 meta-data

앱이 어떤 제어 신호를 쓸지는 코드가 아니라 설치 시 고정된 매니페스트에 선언한다. AgVehicleService는 attach 시점에 PackageManager로 이 값을 직접 읽는다. 앱이 AIDL 파라미터로 "나는 이 신호를 쓴다"고 자기 신고하는 경로는 없다.

meta-data 키필수/선택의미
farm.agmo.vehicle.USES_CONTROL필수쉼표 구분 제어 신호 키 목록. 실행 중인 다른 앱과 겹치면 (동일 tier일 때) 실행 자체가 차단된다.
farm.agmo.vehicle.OPT_CONTROL선택쉼표 구분 제어 신호 키 목록. 겹쳐도 실행을 막지 않고 그 기능만 degrade(비활성)된다.
farm.agmo.vehicle.CONFLICT_WITH선택신호와 무관하게 무조건 공존 불가한 앱 패키지명(쉼표 구분). 어느 쪽이 선언했든 양방향으로 차단된다.
farm.agmo.vehicle.CONTROL_TIER선택값이 정확히 "SYSTEM"일 때만 Tier를 하강시킨다. 그 외/생략은 기본값 APP.
 
<meta-data
    android:name="farm.agmo.vehicle.USES_CONTROL"
    android:value="HITCH_CMD,PTO_CMD" />
<meta-data
    android:name="farm.agmo.vehicle.OPT_CONTROL"
    android:value="AUTO_LIFT_CMD" />
<meta-data
    android:name="farm.agmo.vehicle.CONFLICT_WITH"
    android:value="com.example.othercontrolapp" />
<!-- 생략 시 기본값 APP. "SYSTEM"만 인식 -->
<meta-data
    android:name="farm.agmo.vehicle.CONTROL_TIER"
    android:value="SYSTEM" />
게이트 acquire 시점의 자격 검사는 문자열 정확 일치다. Entitlements.canControl(uid, key)는 key ∈ usesControl ∪ optControl일 때만 true를 반환한다. 매니페스트에 없는 신호는 시도 자체가 거부된다(토큰 0). 단 SAFETY 계층은 플랫폼 내부 경로 전용이라 이 게이트를 거치지 않는다.
코드 선언 불가 자격은 항상 PackageManager로만 읽는다(AgVehicleService.declOf). AIDL 파라미터로 받으면 앱이 런타임에 자기 자신을 자격 있다고 신고하는 셈이라 게이트가 무력화된다. 매니페스트는 설치 시 고정돼 uid와 같은 신뢰 근거를 가진다.
실행 우선순위

실행레벨 Tier

실행레벨 Tier는 제어권을 누가 누구를 선점(preempt)할 수 있는지를 정하는 3계층이다. Tier.rank가 클수록 우선순위가 높다. 순서는 SYSTEM(0) < APP(1) < SAFETY(2)이다. outranks(other)는 rank > other.rank일 때 참이며, 즉 엄격히 상위일 때만 선점할 수 있다. 동급끼리는 선점할 수 없다.

SAFETY플랫폼 전용비상 정지·인터록 등. 매니페스트로 도달 불가 — 앱은 이 등급을 자칭할 수 없다.
APP기본값CONTROL_TIER 미선언 시 이 등급. 같은 제어 신호를 쥔 앱끼리는 동시 실행이 차단된다.
SYSTEM편의 자동화예: 홈 AutoLift. 최하위 — 앱이 언제든 선점해 쥘 수 있다.

AppDeclaration.tier: Tier = Tier.APP이 기본값이고, 매니페스트에서 CONTROL_TIER="SYSTEM"일 때만 하강을 허용한다. 자진 상승은 없다. 스스로를 SAFETY로 선언하는 경로 자체가 없고, 재획득(같은 uid의 재acquire)도 최초 획득 tier를 그대로 유지한다.

SAFETY는 매니페스트로 도달할 수 없다. declOf는 값이 정확히 "SYSTEM"일 때만 하강시키고, 그 외 전부 APP으로 상한을 건다. 플랫폼 전용 승격은 매니페스트를 거치지 않는 별도 내부 경로에서만 이뤄진다.
방어 이중화

이중 방어 — 정적 + 런타임

같은 제어 신호를 두 앱이 동시에 쥐는 상황은 실행 시작 시점(정적)과 제어 요청 시점(런타임) 두 곳에서 각각 막는다. 하나가 뚫려도 다른 하나가 버스에 명령 주체가 둘 이상 서지 않도록 보장한다.

단계시점판정
1차 · ConflictCheckerattach (실행 시작), 정적CONFLICT_WITH 겹침 → 무조건 차단. 필수(usesControl) 신호 겹침 → 같은 tier일 때만 정적 차단(tier가 다르면 런타임 선점이 단일 제어권을 보장하므로 실행 공존을 허용). 선택(optControl) 신호 겹침 → 차단 없이 degrade만.
2차 · ControlArbiteracquire (제어 요청), 런타임동급 보유 중 → Denied. 엄격히 상위만 선점 가능 — StopFirst(안전값 선행) → ReleaseOld(구 세션 반납) → Acquire(신규 획득) 3스텝. 선점당한 앱은 onControlLost 콜백을 수신한다.
예시 홈 AutoLift(SYSTEM)가 히치를 자동 제어 중일 때, 대시보드 전용 제어앱(APP, 기본값)이 같은 히치 신호로 실행된다. tier가 달라 ConflictChecker는 공존을 허용한다. 이후 이 앱이 acquire하면 APP이 SYSTEM을 outranks하므로 ControlArbiter가 선점을 허가하며, AutoLift에 안전값이 먼저 적용되고 구 세션이 반납된 후 앱이 새 세션을 획득한다. AutoLift는 onControlLost를 받고, 앱이 release하면 SYSTEM이 다시 빈 신호를 acquire해 자동화를 재개할 수 있다.
구현 매핑

AgVehicleService 매핑

CONTROL_TIER 파싱은 declOf()에서 이뤄진다. 값이 정확히 "SYSTEM"일 때만 하강하며, 그 외 또는 미선언인 경우는 APP이다. SAFETY는 이 경로로 도달할 수 없다.

val tier = if (meta?.getString(MD_TIER)?.trim()?.uppercase() == "SYSTEM")
    Tier.SYSTEM else Tier.APP

바인더의 acquire는 이렇게 파싱된 선언 tier를 그대로 PolicyEngine.acquire에 넘긴다.

override fun acquire(key: String): Long {
    val uid = Binder.getCallingUid()
    val grant = engine.acquire(uid, key, declOf(uid).tier) ?: return 0L
    grant.preemptedUid?.takeIf { it != uid }?.let { prev ->
        clients[prev]?.cb?.runCatching { onControlLost(key) }
    }
    return grant.token
}
구 버전은 acquire가 tier를 APP으로 하드코딩했다. 지금은 매니페스트 선언 tier(declOf(uid).tier)를 그대로 쓴다. SAFETY가 이 경로로 나올 수 없는 이유는 declOf 자체가 "SYSTEM" 외에는 전부 APP으로 상한을 걸기 때문이다.
